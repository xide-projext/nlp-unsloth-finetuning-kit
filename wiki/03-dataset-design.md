# 03. 데이터셋 설계 — 결과의 절반 이상

> 데이터의 **품질과 양**이 파인튜닝 결과를 그대로 반영합니다. 노이즈는 그대로 학습됩니다.
> 👉 형식을 직접 보려면 [`tutorial/03_tokenization_and_dataset.py`](../tutorial/03_tokenization_and_dataset.py).

---

## Unsloth가 요구하는 3가지

1. **토큰화 가능한 텍스트** — 모델은 글자가 아니라 정수 ID를 먹습니다(토큰화).
2. **role 태깅** — 각 발화가 `user`인지 `assistant`인지 명시.
3. **chat template 정렬** — 베이스 모델마다 대화를 감싸는 특수 토큰 형식이 다름. 그 형식에 맞춰야 함.

---

## 표준 형식 (instruction / chat)

가장 흔한 형태는 `messages` 리스트. JSONL(한 줄 = 학습 샘플 1개)로 저장합니다.

```json
{"messages": [
  {"role": "user", "content": "LoRA가 뭐야?"},
  {"role": "assistant", "content": "원본 가중치는 얼리고 얇은 두 행렬만 학습하는 경량 파인튜닝 기법입니다."}
]}
```

가장 단순한 형태는 **question·answer 두 컬럼**(CSV)도 가능:

| question | answer |
|---|---|
| 연세대 자연어정보분석이 뭐야? | NLP 이론과 LLM 파인튜닝 실습을 다루는 강의입니다. |

chat template을 적용하면 모델이 실제로 보는 문자열은 이런 식(형식은 모델마다 다름):
```
<|user|>
LoRA가 뭐야?
<|assistant|>
원본 가중치는 얼리고 ...
<|end|>
```

---

## Data Recipes — 데이터가 없을 때

Unsloth Studio의 **Data Recipes** 기능은 **PDF·CSV·JSON·DOCX·TXT**를 자동으로 학습용
데이터셋으로 변환해 줍니다. 도메인 문서(강의자료, 매뉴얼, FAQ 등)만 있으면 거기서 Q&A 쌍을
뽑아낼 수 있어, "데이터셋 구하기"가 막혔을 때 출발점이 됩니다.

---

## 품질 체크리스트

- [ ] 모든 샘플이 **토큰화 가능한 텍스트**인가 (깨진 인코딩·바이너리 없음)
- [ ] `user` / `assistant` **role이 정확히** 태깅됐는가
- [ ] 베이스 모델의 **chat template**과 형식이 맞는가
- [ ] 답변이 **정확하고 일관된 스타일**인가 (모델이 그대로 흉내 냄)
- [ ] **중복·모순** 샘플이 없는가
- [ ] 학습/검증 **분리**가 되어 있는가 (과적합 판단용)
- [ ] 양보다 **품질** 우선 — 깨끗한 수백 개 > 지저분한 수만 개

## 양은 얼마나?

| 목적 | 대략적 규모 |
|---|---|
| 말투·형식 흉내 (스타일 튜닝) | 수백 ~ 1천 샘플 |
| 특정 도메인 지식 주입 | 수천 ~ 수만 샘플 |
| 과제 데모(소규모 PoC) | **300–1,000개면 충분** |

> 기말 프로젝트라면 **잘 정제된 300–1,000개**로 시작해, 베이스 대비 변화가 보이는지부터 확인하는
> 게 현실적입니다. 규모를 키우는 건 그다음.

---

## 도메인 정하기 (프로젝트 주제)

"무엇에 특화시킬지"가 곧 데이터셋 주제입니다. 좋은 후보:

- **말투/페르소나** — 특정 캐릭터·존댓말 레벨·브랜드 톤으로 답하기 (소규모로도 효과 큼)
- **도메인 Q&A** — 강의자료·사규·제품 매뉴얼 기반 질의응답 (Data Recipes로 PDF→데이터)
- **포맷 강제** — 항상 JSON/표/특정 템플릿으로 답하기
- **분류·태깅** — 입력을 정해진 라벨로 분류

도메인이 정해지면 [`02-hyperparameters.md`](02-hyperparameters.md)에 맞춰 세팅을 구체화하세요.

## 📄 더 읽을 거리 — 대표 논문

### 데이터 "양보다 질" — 본 섹션의 핵심 주장의 근거

**① LIMA: Less Is More for Alignment (Zhou et al., 2023)**
- **설명 방식**: *Superficial Alignment Hypothesis* — *"모델의 지식·능력은 거의 전부 사전학습에서
  배우고, 정렬(파인튜닝)은 사용자와 상호작용할 때 *어떤 형식·말투*를 쓸지를 가르칠 뿐"*이라는 가설.
  그렇다면 적은 데이터로도 충분하다는 따름정리를 실험으로 검증.
- **직접 예시**: LLaMa **65B를 단 1,000개**의 정제된 예시로 SFT(강화학습 없이) → 사람 평가에서
  GPT-4 대비 **43%**, Bard 대비 **58%**, davinci-003 대비 **65%** 선호. 여행 일정·대체역사 같은
  복잡한 형식을 소수 예시만으로 학습.
- **의미**: 본 섹션의 *"깨끗한 수백~천 개면 충분"* 의 직접 근거.
- 🔗 <https://arxiv.org/abs/2305.11206>

**② InstructGPT (Ouyang et al., 2022)** — 정렬 > 크기
- **설명 방식**: instruction tuning + RLHF(사람 피드백 강화학습)로 '사람이 원하는 답'에 정렬하는
  3단계(SFT→보상모델→PPO) 파이프라인.
- **직접 예시**: 라벨러가 **1.3B InstructGPT의 출력을 175B GPT-3보다 선호** — 잘 정렬된 작은 모델이
  큰 모델을 이길 수 있다. (학생 환경의 작은 모델에 희망적인 결과.)
- 🔗 <https://arxiv.org/abs/2203.02155>

**③ Self-Instruct (Wang et al., 2022)** — 데이터가 없을 때
- **설명 방식**: 모델 *스스로* instruction을 생성해 학습 데이터를 부트스트랩(seed → 생성 → 필터 →
  반복). Studio의 Data Recipes로 문서에서 데이터를 만드는 발상과 통함.
- **직접 예시**: **175개 seed 태스크 → 약 52K개 합성 instruction** 생성, 그것으로 튜닝해 큰 폭 향상.
- 🔗 <https://arxiv.org/abs/2212.10560>

### 토큰화 — 데이터가 모델에 들어가기 전 단계

**④ Neural MT of Rare Words with Subword Units / BPE (Sennrich et al., 2016)**
- **설명 방식**: 번역은 *열린 어휘(open-vocabulary)* 문제라는 직관 — 이름은 글자 복사로, 합성어는
  부분 조합으로 표현 가능하니, 압축 알고리즘 **BPE**를 단어 분절에 차용해 희귀어를 서브워드로 쪼갠다.
- **직접 예시**: 빈도 높은 글자쌍을 반복 병합 → 'est','low' 같은 서브워드 생성(**실습 3-(a)와 동일**).
  WMT15 영-독 +1.1, 영-러 +1.3 BLEU.
- 🔗 <https://arxiv.org/abs/1508.07909>

**⑤ SentencePiece (Kudo & Richardson, 2018)**
- **설명 방식**: 공백까지 `▁` 한 기호로 포함해 raw 텍스트를 **언어 독립적**으로 분절(사전 토큰화
  불필요). 띄어쓰기가 불규칙한 **한국어**에 특히 유용 — 현대 LLM 토크나이저의 사실상 표준.
- 🔗 <https://arxiv.org/abs/1808.06226>
