# 03. 데이터셋 설계 — 결과의 절반 이상

> 데이터의 **품질과 양**이 파인튜닝 결과를 그대로 반영합니다. 노이즈는 그대로 학습됩니다.
> 👉 형식을 직접 보려면 [`tutorial/03_tokenization_and_dataset.py`](../tutorial/03_tokenization_and_dataset.py).

---

## Unsloth가 요구하는 3가지

1. **토큰화 가능한 텍스트** — 모델은 글자가 아니라 정수 ID를 먹습니다(토큰화).
2. **role 태깅** — 각 발화가 `user`인지 `assistant`인지 명시.
3. **chat template 정렬** — 베이스 모델마다 대화를 감싸는 특수 토큰 형식이 다름. 그 형식에 맞춰야 함.

---

## 표준 형식 (instruction / chat)

가장 흔한 형태는 `messages` 리스트. JSONL(한 줄 = 학습 샘플 1개)로 저장합니다.

```json
{"messages": [
  {"role": "user", "content": "LoRA가 뭐야?"},
  {"role": "assistant", "content": "원본 가중치는 얼리고 얇은 두 행렬만 학습하는 경량 파인튜닝 기법입니다."}
]}
```

가장 단순한 형태는 **question·answer 두 컬럼**(CSV)도 가능:

| question | answer |
|---|---|
| 연세대 자연어정보분석이 뭐야? | NLP 이론과 LLM 파인튜닝 실습을 다루는 강의입니다. |

chat template을 적용하면 모델이 실제로 보는 문자열은 이런 식(형식은 모델마다 다름):
```
<|user|>
LoRA가 뭐야?
<|assistant|>
원본 가중치는 얼리고 ...
<|end|>
```

---

## Data Recipes — 데이터가 없을 때

Unsloth Studio의 **Data Recipes** 기능은 **PDF·CSV·JSON·DOCX·TXT**를 자동으로 학습용
데이터셋으로 변환해 줍니다. 도메인 문서(강의자료, 매뉴얼, FAQ 등)만 있으면 거기서 Q&A 쌍을
뽑아낼 수 있어, "데이터셋 구하기"가 막혔을 때 출발점이 됩니다.

---

## 품질 체크리스트

- [ ] 모든 샘플이 **토큰화 가능한 텍스트**인가 (깨진 인코딩·바이너리 없음)
- [ ] `user` / `assistant` **role이 정확히** 태깅됐는가
- [ ] 베이스 모델의 **chat template**과 형식이 맞는가
- [ ] 답변이 **정확하고 일관된 스타일**인가 (모델이 그대로 흉내 냄)
- [ ] **중복·모순** 샘플이 없는가
- [ ] 학습/검증 **분리**가 되어 있는가 (과적합 판단용)
- [ ] 양보다 **품질** 우선 — 깨끗한 수백 개 > 지저분한 수만 개

---

## 양은 얼마나?

| 목적 | 대략적 규모 |
|---|---|
| 말투·형식 흉내 (스타일 튜닝) | 수백 ~ 1천 샘플 |
| 특정 도메인 지식 주입 | 수천 ~ 수만 샘플 |
| 과제 데모(소규모 PoC) | **300–1,000개면 충분** |

> 기말 프로젝트라면 **잘 정제된 300–1,000개**로 시작해, 베이스 대비 변화가 보이는지부터 확인하는
> 게 현실적입니다. 규모를 키우는 건 그다음.

---

## 도메인 정하기 (프로젝트 주제)

"무엇에 특화시킬지"가 곧 데이터셋 주제입니다. 좋은 후보:

- **말투/페르소나** — 특정 캐릭터·존댓말 레벨·브랜드 톤으로 답하기 (소규모로도 효과 큼)
- **도메인 Q&A** — 강의자료·사규·제품 매뉴얼 기반 질의응답 (Data Recipes로 PDF→데이터)
- **포맷 강제** — 항상 JSON/표/특정 템플릿으로 답하기
- **분류·태깅** — 입력을 정해진 라벨로 분류

도메인이 정해지면 [`02-hyperparameters.md`](02-hyperparameters.md)에 맞춰 세팅을 구체화하세요.
